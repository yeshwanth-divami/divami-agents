# CR-001 — Add brownfield support — Daksh currently assumes greenfield only

**Date:** 2026-03-29
**Tasks affected:** inferred: see ## Change Summary
**Raised by:** Yeshwanth
**Tier:** prd

## What was specified

Daksh is a greenfield pipeline. Every stage assumes a clean slate: no prior code,
no prior docs, no inherited conventions, no existing backlog. `init` scaffolds empty
directories; `onboard` asks "what are you building?" as if nothing exists yet.

## What reality showed

Divami teams need to adopt Daksh on existing client repos — projects that already
have code, docs, branch conventions, and partially-scoped backlogs. The current
`init` and `onboard` flows have no path for importing prior state. Running Daksh
on an existing repo produces documentation describing a fiction (the greenfield
plan) rather than the reality (what exists and what needs to change).

Full gap analysis in
[brownfield-deficiencies.md](../../../conversations/client/brownfield-deficiencies.md):
16 deficiencies traced to one root cause — Daksh has no first-class model of
inherited state.

## Impact

Brownfield support is blocked for the majority of Divami engagements. Most new
Daksh adoptions will be retrofits, not clean starts. Without this module, the
pipeline becomes ceremony rather than leverage on those projects.

## Proposed resolution

Add BROWNFIELD as a new module. Full spec via `/daksh prd BROWNFIELD` after approval.
The design doc above covers the 5 architectural moves required.
