# CR-001 — Impl preflight checks module-wide dependencies instead of task-specific ones

**Date:** 2026-03-29
**Tasks affected:** inferred: TASK-PREFLIGHT-002, TASK-PREFLIGHT-003, TASK-PREFLIGHT-004
**Raised by:** Yeshwanth
**Tier:** trd

## What was specified
`/daksh preflight impl MODULE` should verify that the task being started has its declared dependencies done. The TRD says the check reads from the task's `Depends on` field, but the script interface only accepts stage and module.

## What reality showed
The implementation in `scripts/preflight.py` interpreted that gap as a module-wide dependency check. Starting `TASK-PREFLIGHT-003` was blocked by unfinished `TASK-PREFLIGHT-004`, even though 003 does not depend on 004.

## Impact
Stage 50 preflight can block valid task starts, the TRD is ambiguous about scope, and handbook guidance will be wrong if it documents the current behavior as intended.

## Proposed resolution
Update the PREFLIGHT PRD/TRD and `commands/impl/CONTEXT.md` so impl preflight is task-scoped, not module-scoped. Pass the active TASK-ID through the impl-start flow into preflight, then update `scripts/preflight.py`, tests, and `handbook/developer.md` to match.

## Change Summary

### Changes to trd.md
- §Preflight Logic: narrow scope from module-wide to task-specific dependency checking
- §API Surface: add optional `task_id` parameter to `impl_checks()`
